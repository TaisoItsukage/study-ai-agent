# AI業界用語・トレンド - 体系的教材

## はじめに

AI業界は急速に進化しており、新しい用語やトレンドが次々と登場します。本教材では、AI業界の専門用語と最新トレンドを体系的に学び、専門的な会話に対応できる知識を身につけることを目指します。

---

## 第1章: 生成AI（Generative AI）

### 1.1 生成AIとは

**生成AI**は、テキスト・画像・音声・動画などの新しいコンテンツを創り出すAIの総称です。

| カテゴリ | 代表的なサービス |
|---------|--------------|
| テキスト生成 | ChatGPT、Claude、Gemini |
| 画像生成 | Midjourney、DALL-E、Stable Diffusion |
| 動画生成 | Sora（OpenAI） |
| 音声認識・合成 | Whisper（OpenAI） |

### 1.2 生成AIの仕組み

生成AIは、大量のデータから学習したパターンや特徴を組み合わせて、新しいコンテンツを生成します。「創造」というより「**再構成**」に近い仕組みです。

### 1.3 重要な概念

#### ハルシネーション（幻覚）
AIが**事実に基づかない情報を自信を持って生成**してしまう現象。RAG（外部データ連携）で抑制可能。

#### トークン
LLMがテキストを処理する際の最小単位。単語や単語の一部。日本語は1文字が複数トークンになることが多い。

#### 温度（Temperature）パラメータ
出力のランダム性・創造性を制御するパラメータ。

| 温度 | 特徴 | 適した用途 |
|-----|------|----------|
| 低い（0.0〜0.3） | 一貫性が高い、決定的 | 正確性重視の業務 |
| 中程度（0.5〜0.7） | バランス | 一般的な対話 |
| 高い（0.8〜1.0） | 創造的だが予測しにくい | クリエイティブ作業 |

### 1.4 生成AIの活用と注意点

**適した用途**:
- 文章・記事の作成支援
- プログラムコードの生成
- 画像・デザインの生成
- 要約・翻訳

**注意点**:
- 確率的に出力を生成するため、「確定的な予測」には向かない
- 著作権に注意（学習データや生成物）
- 商用利用時は利用規約を確認

## 第2章: GPT・LLM・Transformer

### 2.1 基本用語

| 用語 | 正式名称 | 説明 |
|------|---------|------|
| GPT | Generative Pre-trained Transformer | OpenAIが開発した大規模言語モデルの名称 |
| LLM | Large Language Model（大規模言語モデル） | 膨大なテキストデータで学習した言語AI |
| Transformer | - | 現代のLLMの基盤となるニューラルネットワークアーキテクチャ（2017年Google発表） |

### 2.2 Transformer とAttention機構

**Attention機構**は、入力テキストの各部分の関連性を計算し、文脈に応じて重要な情報に「注意」を向ける仕組みです。これによりLLMは長い文章でも文脈を正確に理解できます。

### 2.3 パラメータ数

パラメータ数はLLMの「脳の大きさ」のようなもので、多いほど高性能な傾向がありますが、**計算リソースとコストも増加**します。

| モデル | パラメータ数（推定） |
|-------|-----------------|
| GPT-3 | 1,750億 |
| GPT-4 | 1兆以上 |
| Llama 2 | 70億〜700億 |

### 2.4 コンテキストウィンドウ

LLMが一度に処理できるトークン数の上限。

| モデル | コンテキストウィンドウ |
|-------|-------------------|
| GPT-4 | 最大128Kトークン |
| Claude 3 | 200Kトークン |

長い文書を一度に分析できるため、契約書や報告書の要約に適しています。

### 2.5 学習プロセス

```
事前学習（Pre-training）→ ファインチューニング → RLHF
```

| プロセス | 説明 |
|---------|------|
| 事前学習 | 大量の汎用データで基礎能力を学習 |
| ファインチューニング | 特定タスクや業界向けに追加学習 |
| RLHF | 人間のフィードバックを使ってAIの回答品質を向上 |

### 2.6 重要な概念

#### エンベディング（Embedding）
テキストや画像を意味を保持した数値ベクトル（数字の配列）に変換する技術。類似した意味のテキストは近いベクトルになる。

#### ベクトルデータベース
エンベディングを効率的に保存・検索するためのデータベース。Pinecone、Weaviate、Chromaなど。

#### セマンティック検索
キーワードの一致ではなく、文章の「意味」を理解して関連する結果を返す検索。

### 2.7 プロンプト技法

| 技法 | 説明 |
|------|------|
| Zero-shot Learning | 例を与えずに指示だけでタスクを実行させる |
| Few-shot Learning | 数個の例を示すだけでタスクを実行させる |
| Chain-of-Thought（CoT） | AIに思考過程を段階的に示させることで推論精度を向上させる |

## 第3章: RAG・ファインチューニング

### 3.1 RAG（Retrieval-Augmented Generation）

RAGは、LLMが回答する際に**外部のデータベースやドキュメントから関連情報を検索**（Retrieval）し、それを参考にして回答を生成（Generation）する手法です。

#### RAGの構成要素

```
ユーザー質問 → リトリーバー → ベクトルDB → 関連文書 → LLM → 回答
```

| コンポーネント | 役割 |
|--------------|------|
| リトリーバー | 質問に関連する文書を検索して取得 |
| ベクトルデータベース | エンベディング化された文書を保存 |
| LLM | 検索結果を基に回答を生成 |

#### RAGのメリット
- 最新の社内情報を反映した回答ができる
- ハルシネーションを抑制できる
- 学習データになかった情報に対応できる

### 3.2 RAGの重要な概念

| 概念 | 説明 |
|------|------|
| チャンキング | 長い文書を適切なサイズに分割する処理 |
| ハイブリッド検索 | セマンティック検索とキーワード検索の組み合わせ |
| リランキング | 検索結果をより関連性の高い順に並べ替える |
| 知識グラフ | 情報間の関係性を構造化して保存する技術 |

### 3.3 ファインチューニング

**ファインチューニング**は、汎用的なLLMを特定の業界・業務に適したものにカスタマイズする追加学習です。

#### 必要なもの
- 目的に合った質の高い**学習データセット**
- 学習を行うための**計算リソース**（GPUなど）

#### 適したケース
- 特定の文体や専門用語を一貫して使いたい場合
- ブランドトーンに合わせた回答スタイルを学習させたい場合

### 3.4 RAGとファインチューニングの比較

| 観点 | RAG | ファインチューニング |
|------|-----|------------------|
| 仕組み | 外部データを参照 | モデル自体を再学習 |
| 情報更新 | データベース更新で対応 | 再学習が必要 |
| コスト | 初期構築は低め | 学習コストがかかる |
| 適した場面 | 情報の更新が頻繁な場合 | 回答スタイルや専門知識を定着させたい場合 |

### 3.5 ファインチューニングの注意点

#### LoRA（Low-Rank Adaptation）
モデル全体ではなく一部のパラメータのみを学習することで、少ないリソースで効率的にファインチューニングを行う手法。

#### 過学習（Overfitting）
学習データに特化しすぎて汎用性が失われること。適切なデータ量と学習手法で防止。

## 第4章: マルチモーダルAI

### 4.1 マルチモーダルAIとは

**マルチモーダルAI**は、テキストだけでなく、画像、音声、動画など複数の種類（モダリティ）のデータを入力・出力できるAIです。

対応モデル: GPT-4V、Gemini、Claude 3

### 4.2 主な技術

| 技術 | 説明 | 代表例 |
|------|------|-------|
| Vision Language Model（VLM） | 画像とテキストを統合的に理解・生成 | GPT-4V、Claude 3 Vision |
| 音声認識（Speech-to-Text） | 音声をテキストに変換 | Whisper |
| 音声合成（Text-to-Speech） | テキストを自然な音声に変換 | - |
| 動画生成AI | テキストから動画を生成 | Sora |
| CLIP | 画像とテキストを同じ空間で表現 | OpenAI CLIP |

### 4.3 活用例

- 写真を見せて「この製品の特徴を説明して」と質問
- 商品写真から説明文を自動生成
- PDFやスキャン画像から情報を抽出・分析
- 会議の音声を文字起こしして要約
- カスタマーサポートの電話対応自動化

### 4.4 注意点

マルチモーダルAIは、画像や音声の処理が加わるため、**テキストのみの処理より計算コストが高くなる**傾向があります。コストと効果のバランスを見て活用しましょう。

## 第5章: オンプレミス vs クラウド

### 5.1 比較

| 観点 | オンプレミス | クラウド |
|------|------------|---------|
| 定義 | 自社設備で運用 | 外部サービスを利用 |
| 初期費用 | 高い | 低い |
| 運用費用 | 固定的 | 従量課金 |
| セキュリティ | データを社外に出さない | ベンダーに依存 |
| スケーラビリティ | 限定的 | 柔軟 |
| 運用負荷 | 高い | 低い |
| 最新機能 | 更新に時間 | すぐに利用可能 |

### 5.2 選択の指針

**オンプレミスが適するケース**:
- 機密性の高いデータを扱う（金融・医療・官公庁）
- データを外部に出せない規制がある
- 低レイテンシーが必要

**クラウドが適するケース**:
- 初期投資を抑えたい
- スケーラブルに利用したい
- まずは小規模から始めたい

### 5.3 ハイブリッド構成

| 構成 | 説明 |
|------|------|
| ハイブリッドクラウド | オンプレミスとクラウドを組み合わせ |
| プライベートクラウド | 特定の組織専用に構築されたクラウド環境 |
| マルチクラウド | 複数のクラウドベンダーを併用 |

### 5.4 重要な概念

#### レイテンシー（遅延）
リアルタイム対話、自動運転、製造ラインの制御など、即座の応答が必要な場面で重要。

#### SLA（Service Level Agreement）
サービスの稼働率（例：99.9%）、応答時間、障害時の対応などを定めた品質保証契約。

## 第6章: エッジAI

### 6.1 エッジAIとは

**エッジAI**は、クラウドではなく、カメラ、センサー、スマートフォンなどのエッジデバイス上でAI処理を行う技術です。

### 6.2 メリットと限界

| メリット | 限界 |
|---------|------|
| 低遅延でリアルタイム処理が可能 | デバイスの計算能力に制限 |
| 通信コストを削減できる | 大規模モデルの実行には不向き |
| プライバシーを保護しやすい | - |

### 6.3 活用例

- 工場のリアルタイム異常検知
- スマートカメラでの人物検知
- IoTセンサーでのデータ分析

### 6.4 最適化技術

| 技術 | 説明 |
|------|------|
| TinyML | 極めて低消費電力のデバイスで動作する機械学習 |
| 量子化（Quantization） | モデルの数値精度を下げてサイズと計算量を削減 |
| モデル圧縮 | モデルサイズを小さくして効率化 |

### 6.5 5Gとの組み合わせ

5Gの低遅延・大容量通信により、エッジでの処理とクラウドでの高度な分析をリアルタイムに連携できます。

## 第7章: MLOps

### 7.1 MLOpsとは

**MLOps**は、機械学習モデルの開発から本番運用、継続的な改善までを効率化・自動化するための手法やツールの総称です。DevOpsの機械学習版と言えます。

### 7.2 MLOpsが重要な理由

AIモデルは、データの変化（ドリフト）により性能が低下するため、継続的な監視と更新が必要です。

### 7.3 重要な概念

| 概念 | 説明 |
|------|------|
| データドリフト | 時間とともに実世界のデータ分布が変化する現象 |
| モデルレジストリ | 学習済みモデルのバージョン管理と保存 |
| Feature Store | 機械学習の特徴量を一元管理・再利用するシステム |
| CI/CD | モデルの継続的な統合とデプロイを自動化する仕組み |
| A/Bテスト | 新旧モデルを一部のユーザーで比較評価 |

### 7.4 MLOpsの流れ

```
データ収集 → 前処理 → 学習 → 評価 → デプロイ → 監視 → 再学習
     ↑__________________________________|
```

## 第8章: AutoML

### 8.1 AutoMLとは

**AutoML**は、データ準備、特徴量エンジニアリング、モデル選択、ハイパーパラメータ調整など、機械学習の構築プロセスを自動化する技術です。

### 8.2 メリットと限界

| メリット | 限界 |
|---------|------|
| 専門知識がなくてもAIモデルを構築できる | 複雑なカスタマイズには専門家が必要 |
| 開発時間を短縮できる | 汎用的なアプローチに留まる |
| PoCを素早く実施できる | - |

### 8.3 主要なAutoMLサービス

| サービス | 提供元 |
|---------|-------|
| Google Cloud AutoML | Google |
| Azure ML | Microsoft |
| Amazon SageMaker | Amazon |

### 8.4 ノーコード/ローコードAI

AutoMLは、コーディングなしでAIモデルを構築できるため、**ノーコード/ローコードAI開発**の中核技術です。プログラミングスキルがなくても、データを用意するだけでAIモデルを構築できます。

## 第9章: Foundation Model・AGI

### 9.1 Foundation Model（基盤モデル）

**Foundation Model**は、大規模データで事前学習され、ファインチューニングやプロンプトで様々なタスクに適用できる汎用モデルです。

例: GPT-4、Claude、Gemini、Llama

### 9.2 AGI（汎用人工知能）

**AGI（Artificial General Intelligence）** は、特定タスクだけでなく、人間のように様々な知的タスクを遂行できるAIを指します。

現在のAI: **特化型AI**（特定タスクに優れるが、AGIではない）

### 9.3 現在のLLMとAGIの違い

| 観点 | 現在のLLM | AGI |
|------|----------|-----|
| 得意分野 | 言語タスク | あらゆる知的タスク |
| 理解 | パターン認識 | 真の理解 |
| 学習 | 事前に学習 | 自律的に学習 |
| 常識推論 | 限定的 | 人間レベル |

### 9.4 重要な概念

| 概念 | 説明 |
|------|------|
| スケーリング則 | モデルサイズ、データ量、計算量と性能の関係を示す法則 |
| エマージェント能力 | モデルの規模が一定以上になると突然現れる能力 |
| 知識蒸留 | 大きなモデルの知識を小さなモデルに転移させる技術 |
| Mixture of Experts（MoE） | 入力に応じて異なる専門サブネットワークを選択的に使用するアーキテクチャ |
| AI Safety | AIの意図しない動作や悪用を防ぐ取り組み |

### 10.4 AIバイアスへの対策
